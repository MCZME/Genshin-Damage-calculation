# 伤害计算与修饰符使用规范

本指南规范 `ModifierRecord` 在伤害计算与实体状态中的使用方式，避免审计链缺失、状态泄漏或 UI 还原错误。

## 1. 基本概念

- `ModifierRecord`：修饰符记录条目（`modifier_id`, `source`, `stat`, `value`, `op`）。定义见 `core/systems/contract/modifier.py`。
- 伤害审计链：`DamageContext.audit_trail`，用于**单次伤害**的审计与 UI 还原。
- 实体动态修饰符：`CombatEntity.dynamic_modifiers`，用于**实体持续状态**的属性修饰符与生命周期管理。

## 2. 伤害计算侧（DamageContext）

### 2.1 统一入口

- 必须通过 `DamageContext.add_modifier(source, stat, value, op, audit)` 写入。
- 严禁直接修改 `DamageContext.stats`（会导致审计链缺失且 UI 无法还原）。
- 伤害侧修饰符通常也应封装在 `Effect` 中，但由于伤害是**瞬时事件**，应通过事件注入（如 `EventType.BEFORE_CALCULATE`）在结算前生效。
- 伤害侧效果必须**严格添加判定条件**（例如 `damage.name`、`attack_tag`、`source/target`、元素类型等），避免将效果注入到不匹配的伤害上。

### 2.2 审计策略（[S]/[R]）

- `[R]`（检索项）：可从 `frame_id` 快照唯一还原的面板值，使用 `audit=False`。
- `[S]`（存储项）：瞬时或不可预测的数据（技能倍率、动态 Buff、暴击判定等），使用 `audit=True`。

### 2.3 可修饰字段（标准槽位）

以下字段允许通过 `add_modifier` 注入（即 `[S]` 或 `[R]+[S]`）：

- `技能倍率%`
- `独立乘区%`
- `倍率加值%`
- `固定伤害值加成`
- `伤害加成`（动态增伤部分）
- `暴击乘数`
- `反应加成系数`
- `基础属性`（仅“动态固定值加成”部分，例如班尼特 Q 的固定攻击力）

说明与乘区归属详见 `docs/core/伤害系统.md` 与 `docs/architecture/system/伤害审计系统架构.md`。

### 2.4 可用字段（伤害计算侧）

以下字段会直接影响 `DamageContext` 的合算流程。字段名必须完全匹配：

| 字段名 | 说明 | 作用阶段/乘区 |
| :--- | :--- | :--- |
| {属性名} | 作为缩放属性的最终值（如 `攻击力`、`生命值`） | Foundation / 核心伤害合算 |
| {属性名}技能倍率% | 技能倍率（如 `攻击力技能倍率%`） | Foundation / 倍率区 |
| 独立乘区% | 独立乘区加成 | Synthesis / 倍率区 |
| 倍率加值% | 申鹤/云堇等倍率加值 | Synthesis / 倍率区 |
| 固定伤害值加成 | 固定伤害加算 | Synthesis / 核心区 |
| 伤害加成 | 全伤害增伤（%） | Synthesis / 增伤区 |
| 暴击乘数 | 暴击最终乘数 | Resolution / 暴击区 |
| 反应基础倍率 | 反应基础倍率（如蒸发/融化倍率） | Resolution / 反应区 |
| 反应加成系数 | 精通转化与反应增益 | Resolution / 反应区 |
| 元素精通 | 反应增益计算使用 | Resolution / 反应区 |
| 暴击率 | 暴击判定用（加到面板暴击率） | Resolution / 暴击区 |
| 暴击伤害 | 暴击判定用（加到面板暴伤） | Resolution / 暴击区 |

备注：
- `{属性名}` 与 `{属性名}技能倍率%` 由 `Damage.scaling_stat` 决定，须与缩放属性一致。
- `反应基础倍率` 通常为 `[R]`（由反应结果/面板得出），`反应加成系数` 与 `暴击乘数` 等动态项通常为 `[S]`。

### 2.5 命名规范

- 标准槽位必须使用规范字段名（如 `伤害加成`、`防御区系数`）。
- 多属性缩放倍率必须使用 `{属性名}技能倍率%`（例如 `攻击力技能倍率%`）。
- `source` 必填，用于 UI 溯源与审计展示。

### 2.6 示例（伤害侧修饰符）

```python
class DamageBonusEffect:
    def __init__(self, bonus_ratio: float = 0.4) -> None:
        self.bonus_ratio = bonus_ratio

    def handle_event(self, event: GameEvent):
        if event.event_type != EventType.BEFORE_CALCULATE:
            return

        ctx = event.data.get("damage_context")
        if not ctx:
            return

        # 仅对指定伤害生效，避免误注入
        if ctx.damage.name != "示例伤害名称":
            return

        # 瞬时注入：结算前写入伤害加成，必须进入审计链
        ctx.add_modifier(
            source="示例-动态增伤",
            stat="伤害加成",
            value=self.bonus_ratio * 100.0,
            op="ADD",
            audit=True,
        )
```

## 3. 实体动态修饰符（CombatEntity）

### 3.1 适用场景

- 用于实体的**持续状态**修饰：光环、长期 Buff、场景效果等。
- 不用于“单次伤害结算”的审计链记录。

### 3.2 使用方式

- 实体动态修饰符通常应封装在 `Effect` 类中管理，由效果生命周期统一负责添加与移除。
- 避免在业务逻辑中直接、散落地调用 `entity.add_modifier` / `entity.remove_modifier`。

### 3.2 统一入口与事件

- 使用 `CombatEntity.add_modifier(...)` 写入，返回 `ModifierRecord`。
- 写入与移除会触发事件：
  - `EventType.ON_MODIFIER_ADDED`
  - `EventType.ON_MODIFIER_REMOVED`

### 3.3 生命周期要求

- 实体销毁时会调用 `clear_active_states()`，自动清理 `dynamic_modifiers`。
- 外部系统不得绕过 `remove_modifier` 直接修改列表。

### 3.4 可用字段（实体动态修饰符）

以下字段会被 `AttributeCalculator` 合算时读取。未在此表中的 `stat` 将不会影响最终属性：

| 字段名 | 说明 | 作用范围 |
| :--- | :--- | :--- |
| 攻击力% | 百分比攻击力加成 | 参与 `get_final_atk` 合算 |
| 固定攻击力 | 固定攻击力加成 | 参与 `get_final_atk` 合算 |
| 生命值% | 百分比生命值加成 | 参与 `get_final_hp` 合算 |
| 固定生命值 | 固定生命值加成 | 参与 `get_final_hp` 合算 |
| 防御力% | 百分比防御力加成 | 参与 `get_final_def` 合算 |
| 固定防御力 | 固定防御力加成 | 参与 `get_final_def` 合算 |
| 元素精通 | 元素精通加成 | 参与 `get_final_em` 合算 |
| 暴击率 | 暴击率加成（%） | 参与 `get_final_crit_rate` 合算 |
| 暴击伤害 | 暴击伤害加成（%） | 参与 `get_final_crit_dmg` 合算 |
| 元素充能效率 | 充能效率加成（%） | 参与 `get_final_er` 合算 |
| 治疗加成 | 治疗加成（%） | 参与 `get_final_healing_bonus` 合算 |
| 受治疗加成 | 受治疗加成（%） | 参与 `get_final_incoming_healing_bonus` 合算 |
| 护盾强效 | 护盾强效（%） | 参与 `get_final_shield_strength` 合算 |
| 伤害加成 | 全伤害加成（%） | 参与 `get_final_damage_bonus` 合算 |
| {元素}元素伤害加成 | 元素伤害加成（%），如 `火元素伤害加成` | 参与 `get_final_damage_bonus` 合算 |
| 物理伤害加成 | 物理伤害加成（%） | 参与 `get_final_damage_bonus` 合算 |
| {元素}元素抗性 / 物理元素抗性 | 抗性加成（%），如 `火元素抗性`、`物理元素抗性` | 参与 `get_final_res` 合算 |

备注：
- 实体动态修饰符当前仅在合算中**累加**（`ADD` 语义）。`MULT`/`SET` 暂不生效。
- 元素名与抗性名需与现有属性命名一致，否则不会被识别。

### 3.5 示例（实体侧修饰符）

```python
class AttackBuffEffect:
    def __init__(self, duration_frames: int = 300) -> None:
        self.duration_frames = duration_frames
        self.modifier = None
        self.elapsed = 0

    def on_apply(self, entity):
        self.modifier = entity.add_modifier(
            source="示例-持续增益",
            stat="攻击力%",
            value=20.0,
            op="ADD",
        )

    def on_frame_update(self, entity):
        self.elapsed += 1
        if self.elapsed >= self.duration_frames:
            self.remove(entity)

    def remove(self, entity):
        if self.modifier is not None:
            entity.remove_modifier(self.modifier)
            self.modifier = None
```

## 4. 常见错误

- 直接改写 `stats` 或 `attribute_data`，导致审计链缺失。
- 应为 `[R]` 的面板值仍以 `audit=True` 写入，导致数据库冗余。
- 使用非规范 `stat` 命名，导致审计乘区映射失败或 UI 无法缝合。
- 动态修饰符未及时移除，造成实体死亡后仍保留状态。

## 5. 负面字段使用规范（目标侧）

### 5.1 基本原则

- 负面字段（通常作用于**目标**）应通过 **效果（Effect）** 的属性字段作用于目标实体，而非通过伤害计算字段注入。
- 原则上避免在 `DamageContext` 中使用“减抗/减防”类字段。

### 5.2 示例：减抗

降低火元素抗性 10% 的做法：

1. 创建一个效果（Effect）。
2. 在效果应用阶段向目标实体添加修饰符：`火元素抗性`，`-10`。

### 5.3 示例：减防

减防应作用于 `防御力%`，而不是通过伤害侧字段：

1. 创建一个效果（Effect）。
2. 在效果应用阶段向目标实体添加修饰符：`防御力%`，负值（例如 `-30`）。

### 5.4 特殊字段：无视防御%

- `无视防御%` 属于伤害结算侧的**特殊字段**，不通过实体属性常驻。
- 需要通过 `Effect` 在伤害事件注入阶段（如 `EventType.BEFORE_CALCULATE`）临时写入 `DamageContext`。
