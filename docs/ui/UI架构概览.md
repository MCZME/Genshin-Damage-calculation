# UI 架构概览 (V4.0)

> 本文档描述了仿真引擎 V4.0 的整体 UI 架构设计，重点介绍了状态管理、视图解耦及全局渲染机制。

---

## 1. 整体架构模型 (Architecture Overview)

项目 UI 采用 **“中心化状态 + 局部事件驱动”** 的模型，旨在解决复杂仿真数据带来的高频刷新与界面性能瓶颈。

### 1.1 核心分层
1.  **宿主层 (AppLayout)**: 负责全局导航（战略、战术、分析）、顶部/底部常驻工具栏以及全屏视图切换。
2.  **视图层 (View Layer)**: 各功能模块（如 `StrategicView`, `AnalysisView`）作为独立组件挂载，持有各自的业务逻辑。
3.  **状态层 (AppState / AnalysisState)**: 存储 UI 响应式数据，负责与核心仿真引擎及数据库通信。
4.  **组件层 (Component Layer)**: 最小原子单元（磁贴、输入框、卡片），遵循统一的艺术风格规范。

---

## 2. 通信机制 (Communication)

### 2.1 UI 事件总线 (UIEventBus)
为了避免深层嵌套组件间的属性透传（Props Drilling），系统引入了全局事件总线：
*   **信号解耦**: 状态层通过 `events.notify("event_name")` 发出信号，视图层订阅后进行局部刷新。
*   **异步加载**: 特别适用于分析界面的“下钻”逻辑，允许 UI 先显示加载态，待数据库异步查询结束后自动填充。

### 2.2 脏标记系统 (Dirty Flag System)
为了降低多视图环境下的内存占用，AppLayout 实现了脏标记机制：
*   处于后台的视图（如不可见的战略页）在数据变动时仅标记“Dirty”。
*   仅当用户点击切换至该页面时，才执行全量刷新（Lazy Refresh）。

---

## 3. 渲染引擎与主题 (Theme & Rendering)

### 3.1 统一主题管理器 (GenshinTheme)
所有组件的色彩、边距、字体、动效曲线必须引用 `ui/theme.py`。
*   **V4.0 核心变更**: 引入了 `GOLD_LIGHT` 和 `GOLD_DARK` 常量，支持“Artifact Prime”金属风格渲染。

### 3.2 布局引擎 (Responsive Engine)
*   采用 Flet 的 `ResponsiveRow` 作为主布局底座。
*   磁贴通过 `col` 属性定义其在 12 列栅格中的占比，配合高度锁定策略，实现稳定的网格流。

---

## 4. 关键文件索引
*   `ui/app.py`: 程序入口与生命周期管理。
*   `ui/layout.py`: 主框架布局与导航控制。
*   `ui/states/`: 各业务模块的同步/异步状态机。
*   `ui/components/`: 可复用的原子组件库。
