# 持久化架构技术规范 (V3.0 Architecture)

## 1. 架构总览
本项目采用 **“垂直拆分 (Vertical Partitioning)”** 与 **“多轨投影 (Multi-Track Projection)”** 的混合架构。通过将仿真数据依据逻辑性质分流存储，彻底解决了全量 JSON 快照导致的 I/O 性能瓶颈与数据冗余问题。

### 1.1 核心设计原则
*   **计算与存储解耦**：仿真引擎负责逻辑运算，持久化层通过“投影器”将内存状态映射为数据库记录。
*   **审计为先**：每一笔伤害都必须具备完整的溯源能力，通过唯一 ID 关联 Buff 的生命周期。
*   **冷热分离**：结构化数值列用于统计（热数据），原始 JSON Payload 隔离存储（冷数据）。

---

## 2. 数据流转机制

### 2.1 静态登记自动化 (Discovery-based Registration)
*   **机制**：Worker 端维护已登记实体缓存。
*   **流程**：当快照中出现未见过的 `entity_id` 时，自动提取元数据并填充 `registry` 及其特化表（`characters`/`targets`）。
*   **收益**：核心引擎无需显式调用保存接口，实现“发现即登记”。

### 2.2 动态标识体系 (Identity System)
*   **Modifier ID**: `ModifierRecord` 持有 Session 内唯一的 ID，用于闭合生命周期区间。
*   **Instance ID**: 效果与护盾持有独立 ID，区分同名实例。
*   **Event ID**: 由数据库自增生成，作为审计明细与原始负载的关联锚点。

## 3. 表结构详解

### 3.1 基础管理表
| 表名 | 职能 | 核心字段 |
| :--- | :--- | :--- |
| `simulation_sessions` | 会话索引 | `id`, `config_name`, `total_damage`, `duration_frames` |
| `simulation_configs` | 配置详情 | `session_id`, `config_snapshot` (重型 JSON) |

### 3.2 静态登记体系
| 表名 | 职能 | 关键列 |
| :--- | :--- | :--- |
| `simulation_entity_registry` | 通用身份 | `entity_id`, `entity_type`, `hitbox_radius`, `hitbox_height` |
| `simulation_characters` | 角色面板 | `level`, `constellation`, `base_attributes`, `artifact_sets` |
| `simulation_targets` | 目标抗性 | `level`, `base_defense`, `res_phys`, `res_fire...res_rock` |
| `simulation_entities` | 召唤物归属 | `owner_id`, `created_frame`, `duration` |

## 4. 动态追踪轨道 (Tracking Tracks)

| 轨道类型 | 对应表名 | 记录逻辑 |
| :--- | :--- | :--- |
| **Pulse** | `character_pulses` | 记录角色的坐标、动作 ID、在场状态。 |
| **Jump** | `simulation_state_jumps` | 稀疏存储 HP、ENERGY 的阶梯式跳变。 |
| **Lifecycle** | `modifier_lifecycles` | 记录属性加成 (Modifiers) 的生效与失效区间。 |
| **Effect** | `simulation_effect_lifecycles` | 记录护盾或状态的存续区间及预期持续时间。 |
| **Shield Jump** | `simulation_shield_jumps` | 记录护盾剩余量的实时变动。 |
| **Aura Pulse** | `target_aura_pulses` | 高频采样目标的元素附着状态。 |
| **Mechanism** | `simulation_mechanism_metrics` | 自动采集 `custom_metrics` 中的通用机制指标。 |

### 4.1 通用机制指标轨道 (Mechanism Track)
针对如“气氛值”、“燃点”、“夜魂值”等不断增加的新机制，采用通用的指标映射模式。
*   **采集点**：实体持有的 `custom_metrics: Dict[str, float]`。
*   **存储逻辑**：由投影层进行自动扫描，发现数值跳变时记录。
*   **收益**：无需修改代码即可支持无限量的机制数值持久化。

---

## 5. 离散事件审计体系 (三层架构)
1.  **骨干层 (`simulation_event_log`)**: 记录事件类型、发生帧、发起者。
2.  **特化层 (`event_damage_data`)**: 平铺核心伤害维度 (value, element, tag, crit)。
3.  **明细层 (`event_audit_trail`)**: 逐行记录计算步骤，引用 `modifier_id` 实现溯源。
4.  **备份层 (`event_payloads`)**: 存储原始 JSON 备份。

---

## 6. 查询与还原逻辑
UI 适配器通过以下逻辑重建第 $T$ 帧快照：
1.  **生存判定**：筛选 `registry` 中在 $T$ 帧处于活跃区间的实体。
2.  **状态追溯**：对每个实体，取 `frame_id <= T`且最大的 `pulse` 和 `jump` 记录。
3.  **属性叠加**：取所有覆盖 $T$ 帧的 `lifecycles`，通过 `AttributeCalculator` 逻辑应用到 `base_attributes`。
