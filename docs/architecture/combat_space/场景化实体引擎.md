# 架构设计：场景化实体引擎 (CombatSpace)

## 1. 概述
本设计将战场抽象为一个 **2D 连续坐标空间 (X, Z)**，辅以物理高度 (Y)。在 V2.5 架构中，`CombatSpace` 进化为 **”战斗世界管理器”**，它不仅负责实体的空间检索，还作为逻辑队伍 (`Team`) 的宿主，协调物理感知与逻辑驱动。

## 2. 实体层级架构

### 2.1 基类设计 (V2.5 重构)
```
BaseEntity (core/entities/base_entity.py)
│   ├── 生命周期管理：INIT → ACTIVE → FINISHING → DESTROYED
│   ├── 空间属性：pos, facing, hitbox, faction
│   ├── 帧驱动接口：on_frame_update() → _perform_tick()
│   └── 状态导出：export_state(), export_static_data()
│
├── CombatEntity (战斗实体)
│   │   ├── 元素附着：AuraManager, ICDManager
│   │   ├── 效果管理：active_effects, shield_effects
│   │   ├── 属性修饰符：dynamic_modifiers
│   │   └── 子类：Character, Target, 召唤物等
│
├── AttackEntity (攻击实体)
│   │   ├── 承载一次攻击
│   │   ├── 触发类型：ON_FINISH, ON_HIT_FRAME, ON_PROXIMITY, TRACKING
│   │   └── 索敌模式：CLOSEST, RANDOM, FIXED
│
└── SceneEntity (场景实体)
    │   ├── 长期存在场景中
    │   ├── 区域检测：实体进入/离开/停留
    │   └── 周期触发：tick_interval
```

### 2.2 设计理念
- **BaseEntity** 提供最小化的空间属性（pos, facing, hitbox, faction），所有实体都具备位置和阵营。
- **CombatEntity** 扩展战斗能力（元素附着、效果管理），适用于需要接收伤害或管理状态的实体。
- **AttackEntity** 轻量级攻击载体，不继承 CombatEntity 以避免不必要的复杂性。
- **SceneEntity** 场景效果载体，用于区域检测和周期性逻辑。

## 3. 核心职责
### 3.1 实体管理 (Unified Storage)
- **存储容器**：`_entities: dict[Faction, list[BaseEntity]]` 存储所有非角色实体。
- **角色本体**：由 `Team` 实例持有，空间查询时动态合并。
- **类型兼容**：支持 `BaseEntity` 的所有子类（CombatEntity, AttackEntity, SceneEntity）。

### 3.2 级联驱动 (Cascade Update)
`CombatSpace.on_frame_update` 负责驱动每一帧的演进：
1. **驱动逻辑大脑**：调用 `team.on_frame_update()`，循环更新全队成员的 ASM、技能 CD 及属性审计。
2. **驱动物理实体**：调用 `_entities` 中各实体的 `on_frame_update()`，处理敌人移动、召唤物行为、攻击实体追踪等。
3. **状态清理**：移除生命周期结束或已销毁的实体。

## 4. 空间模型
### 4.1 坐标系统
- **XZ 平面**：用于所有 AOE 形状（圆、矩形）的碰撞判定。
- **Y 轴 (高度)**：用于下落攻击判定。下落攻击技能会通过 `on_frame_update` 修改角色的 Y 坐标，当 Y <= 0 时触发落地冲击。

### 4.2 判定接口
- `get_entities_in_range(origin, radius, faction) -> list[BaseEntity]`：检索指定半径内的所有实体。
- `get_entities_in_box(origin, length, width, facing, faction) -> list[BaseEntity]`：检索矩形范围内的实体。
- `broadcast_damage(attacker, damage)`：核心接口。根据 `AttackConfig` 执行全场扫描，仅 `CombatEntity` 类型实体会接收伤害。

## 5. 与 Team 的协作关系
- **单一来源**：空间通过 `set_team` 建立与队伍的关联。
- **物理隔离**：确保空间中永远只存在”场上”那个物理点，避免后台角色被 AOE 错误命中。
- **驱动唯一性**：角色逻辑仅由 `Team` 驱动一次，物理查询仅作为快照数据读取。
