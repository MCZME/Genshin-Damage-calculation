# Core - 伤害系统 (Damage System)

伤害系统是仿真引擎 V2.4 的核心，负责实现从攻击意图到最终数值结算的全流程逻辑。该系统采用了高度标准化的流水线设计，并深度集成了全流程审计能力。

## 1. 核心概念

### 1.1 Damage 行为载体 (`core.systems.contract.damage.Damage`)
`Damage` 对象封装了攻击的本质物理契约，不持有计算逻辑，仅作为数据容器在系统中流转：
- **倍率向量 (`damage_multiplier`)**: 支持多段伤害（如 3 段攻击）的倍率列表。
- **属性依赖 (`scaling_stat`)**: 定义伤害挂钩的面板属性（如“攻击力”、“生命值”或“固定值”）。
- **元素属性 (`element`)**: 包含元素类型（`Element` 枚举）与元素量（Gauge）。
- **数据包 (`data`)**: 动态存储计算结果及最重要的 `audit_trail`（审计链）。

### 1.2 DamageContext 审计状态机
`DamageContext` 维护了一个预定义的属性字典 `stats`，所有计算步骤必须在此字典内进行增量修改：
- **核心属性 Key**: 
    - 基础属性：`攻击力`, `生命值`, `防御力`, `元素精通`
    - 增益属性：`固定伤害值加成`, `伤害加成`, `暴击率`, `暴击伤害`
    - 乘区系数：`防御区系数`, `抗性区系数`, `反应基础倍率`, `反应加成系数`
- **操作类型**: 
    - `ADD`: 累加（用于面板、增伤、固定值等）。
    - `MULT`: 累乘（用于独立乘区等）。
    - `SET`: 直接赋值（用于重写反应倍率或特定系数）。

## 2. 伤害结算流水线 (DamagePipeline)

### 第一阶段：标签解析与事件发布
1. **解析分类**: 利用 `AttackTagResolver` 将 `attack_tag` 映射为 `NORMAL` (普攻)、`SKILL` (战技) 等逻辑分类集合。
2. **发布 BEFORE_CALCULATE**: 触发该事件，允许所有系统（武器、圣遗物、天赋）通过 `ctx.add_modifier` 注入特定增益。

### 第二阶段：空间广播与碰撞探测
1. **CombatSpace 介入**: 调用场景管理器进行 3D 几何碰撞判定。
2. **高度感知**: 根据实体的 Y 轴坐标实时判断下落攻击的有效性或范围判定。
3. **确定目标**: 建立 `target` 引用并同步触发目标的 ICD（内部冷却）判定。

### 第三阶段：快照与属性注入 (`_snapshot_instant`)
系统根据 `Damage.scaling_stat` 向量执行自动装配：
1. **基础面板**: 从 `AttributeCalculator` 抓取对应的基础属性（如攻击力）并注入 `stats`。
2. **倍率注入**: 将契约中的 `damage_multiplier` 按索引注入为 `[技能契约] {stat}倍率`。
3. **动态增伤汇总**: 自动汇总由 `AttackTagResolver` 匹配到的元素类型及动作分类（普攻、战技等）的总增伤值。

### 第四阶段：反应预处理 (`_preprocess_reaction_stats`)
1. **触发反应**: 若存在 `reaction_results`，则通过 `SET` 操作注入 `反应基础倍率`（如蒸发 1.5x）。
2. **精通修正**: 注入由精通及外部 Buff 提供的 `反应加成系数`。

### 第五阶段：防御与抗性结算 (`_calculate_def_res`)
系统采用标准的分段函数处理环境抗性：
- **防御系数 (Def Coeff)**: 
  $$Coeff_{def} = \frac{5 \times Level_{src} + 500}{Target_{def} + 5 \times Level_{src} + 500}$$
- **抗性系数 (Res Coeff)**:
    - 若 $Res > 75\%$: $1 / (1 + 4 \times Res / 100)$
    - 若 $Res < 0$: $1 - (Res / 2 / 100)$
    - 其他: $1 - Res / 100$

### 第六阶段：最终数值合算 (`_calculate`)

#### A. 常规计算路径 (增幅/强化反应)
```text
伤害 = (基础伤害向量点积 + 固定伤害值加成) 
       * (1 + 伤害加成 / 100) 
       * 暴击乘数 
       * (反应基础倍率 * (1 + 反应加成系数)) 
       * 防御区系数 
       * 抗性区系数
```
*其中点积逻辑：若属性名非“固定值”，则为 `属性值 * (倍率 / 100)`；否则直接累加倍率。*

#### B. 剧变计算路径 (扩散/超载等)
当 `AttackCategory.REACTION` 标签存在时，系统执行独立路径：
$$EM_{bonus} = \frac{16 \times EM}{EM + 2000}$$
$$Final = 等级系数 \times 反应基础系数 \times (1 + EM_{bonus} + 反应特定加成) \times 抗性区系数$$

## 3. 开发规范与审计追踪

1. **计算即审计**: 严禁直接修改 `ctx.stats`。所有变更必须通过 `ctx.add_modifier` 注入，系统会自动分配全局唯一 `modifier_id` 并记录在 `audit_trail` 中。
2. **命名规范**: 审计来源 `source` 应包含具体上下文（如 `[实体面板]`、`反应:OVERLOADED`），以便在 UI 审计图中清晰呈现。
3. **暴击模拟**: 若开启 `emulation.open_critical`，系统会在结算时进行随机判定，成功后向审计链注入 `暴击区` 记录并应用 `(1 + 暴击伤害 / 100)` 乘数。
